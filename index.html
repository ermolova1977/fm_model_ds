1. Цель разработки модели

Цель разработки — создание универсальной финансовой модели девелоперского проекта, которая позволит:

  • моделировать строительство нескольких очередей проекта;
  • учитывать несколько жилых домов внутри каждой очереди;
  • учитывать отдельный договор проектного финансирования (PF) для каждой очереди;
  • рассчитывать ставку кредитования в зависимости от наполнения эскроу-счетов;
  • учитывать временное межэтапное фондирование очередей;
  • автоматически формировать модель для банка (Bank View);
  • автоматически формировать управленческую отчетность девелопера;
  • автоматически формировать финансовые отчеты проекта (ДДС, P&L, Баланс).

Модель должна использоваться для:

  • внутреннего финансового планирования;
  • переговоров с банками;
  • подготовки инвестиционных решений;
  • контроля финансовой устойчивости проекта.

2. Общие требования к модели

2.1 Формат

Microsoft Excel (.xlsx). На первом этапе модель должна работать без макросов.

2.2 Горизонт планирования

120 месяцев (10 лет), с возможностью расширения до 180 месяцев (15 лет).

2.3 Масштабируемость

ПараметрЗначение
Очереди проектадо 5
Дома в очередидо 5
PF договорыотдельный на очередь
СценарииBase / Conservative / Aggressive

3. Архитектура модели

Трёхуровневая архитектура:

3.1 CORE MODEL (Developer View)

Реальная экономика проекта: денежные потоки, межэтапное фондирование, запуск со сдвигом, фактическая структура финансирования. Для внутреннего управления.

3.2 POLICY LAYER

Слой трансформации: корректировка по банковским требованиям, исключение внутренних потоков, подготовка bank-view.

3.3 BANK MODEL

Для представления банку: очищенная структура, покрытие, ковенанты, escrow-модель.

8. Escrow модель

EscrowBalance(t) = EscrowBalance(t-1) + SalesRevenue
Раскрытие эскроу после ввода дома.

9. Расчет ставки PF

Ставка зависит от покрытия: Coverage = (F + 0.5 × F2) / Debt, где F — средства на эскроу, F2 — ипотечные кредиты.

10. Межэтапное финансирование

Временное фондирование: Stage1 → Stage2, Stage2 → Stage3. Эти потоки исключаются из Bank View.

13. Красные флаги (Red Flags)

DSCR < 1.2
Equity ratio < 20%
Coverage < 100%
Debt отрицательный
Sales превышают площадь

14. Графики

Dashboard содержит графики: продажи, строительство, наполнение эскроу, задолженность, ставка PF, денежный поток.

15. Комментарии к показателям

Каждый ключевой показатель должен иметь пояснение. Пример:

DSCR
Показатель способности проекта обслуживать долг. Минимальное значение для банковского финансирования — 1.2.

16. Автоматические выводы

Модель формирует выводы, например: «Проект финансово устойчив. DSCR выше минимального уровня банка. Escrow coverage обеспечивает снижение процентной ставки.»

17. Автоматические рекомендации

Генерация рекомендаций: ускорить продажи, перенести запуск очереди, снизить CAPEX, увеличить equity.

18. Model Audit

Не менее 40 автоматических тестов по группам: Structure, Logic, Debt, Bankability, Integrity.

19. Итоговые показатели

  • IRR проекта
  • IRR девелопера
  • NPV
  • Peak Debt
  • Equity requirement
  • Developer profit

20. Критерий готовности модели

Модель считается выполненной, если:

  • автоматически рассчитывает проект минимум из 3 очередей;
  • формирует ДДС, P&L и Баланс;
  • формирует Bank View;
  • содержит Model Audit Layer;
  • содержит Dashboard с графиками, red flags и рекомендациями.
